## OptiTrack → Mac → Planner 现场适配方案

### 一、当前现场状态

#### Windows / Motive

* Windows 连接 OptiTrack 摄像机并运行 Motive。
* Motive 目标帧率：`360 Hz`。
* Motive Streaming 已开启。
* Local Interface：`192.168.50.1`
* Transmission Type：`Unicast`
* Up Axis：`Z Axis`
* Rigid Bodies：开启
* 当前刚体：

  * 球桌：`table`
  * 乒乓球：`ball`
* `ball` 表面有 7 个反光标记，通常同时可见 3～4 个，目前作为刚体使用。

#### Mac

* Mac 拓展坞网口：`192.168.50.2/24`
* Windows 和 Mac 已经能够互相通信。
* Mac 上有 Planner：`predict_node_0627.py`
* 上次 NatNet 可视化脚本已经丢失，需要重新实现。

---

## 二、职责边界

### Windows / Motive 负责

* 摄像机采集和三维重建。
* `table`、`ball` 刚体识别。
* 刚体名称、Marker 布局、Pivot、Streaming 配置。
* 输出 NatNet 数据。

### Mac 负责

Mac 端是 **只读客户端和适配层**，主要负责：

1. 接收 Windows Motive 的 NatNet 数据。
2. 解析帧号、时间戳和刚体数据。
3. 检查 `table`、`ball` 是否存在且 Tracking Valid。
4. 将 Motive 世界坐标转换成 Planner 所需坐标。
5. 计算球速度。
6. 检查帧率、丢帧、坐标、单位和数据连续性。
7. 通过 ZMQ 输出 Planner 所需消息。
8. 判断现有 Planner 是否能够正常触发预测。

Mac 端 **不能也不应该尝试修改**：

* Motive 的 Streaming 设置；
* Windows 网卡设置；
* 刚体名称；
* Marker 模型；
* 刚体 Pivot；
* Motive 坐标轴。

Mac 如果检测到这些问题，只输出明确的诊断结果和需要在 Windows 执行的修改建议。

## 三、当前代码结构

当前实现位于 `mocap/hope_optitrack_adapter/`，保持最小但清晰的边界：

```text
mocap/hope_optitrack_adapter/
├── natnet_receiver.py          # NatNet Unicast command/data receiver
├── coordinate_adapter.py       # Motive/table frame -> planner frame
├── validation.py               # ball valid/dt/speed checks
├── diagnostics.py              # JSONL + stdout fps/latency/valid/drop
├── ros_receiver.py             # robot NamedPoseArray
├── schema_adapter.py           # m/m/s -> planner mm/mm/s schema
├── zmq_publisher.py            # send_pyobj() publisher
├── config.yaml
└── main.py
```

职责：

```text
natnet_receiver
    连接 Motive 192.168.50.1:1510/1511，解析 MODELDEF / FRAMEOFDATA，
    根据刚体 name 映射 ID，读取 ball rigid body

coordinate_transform
    Motive world → table frame → planner frame

velocity_estimator
    当前在 natnet_receiver 内基于位置和 Motive timestamp 做最小差分估计

diagnostics
    FPS、丢帧、tracking valid、抖动、异常跳变

zmq_publisher
    按 Planner 现有协议发送 ball / robot 消息
```

不修改 Planner，除非确认是 Planner 自身运行环境或协议问题。

## 四、配置文件

建议配置：

```yaml
natnet:
  server_ip: "192.168.50.1"
  local_ip: "192.168.50.2"
  command_port: 1510
  data_port: 1511
  connection_type: "unicast"
  bitstream_version: [4, 1]

ball:
  source: natnet_rigid_body
  rigid_body_name: "ball"

table:
  rigid_body_name: "table"

robot:
  source: motion_capture_tracking_named_pose
  topic: "/motion_capture_tracking/poses"
  rigid_body_name: "P1"
  mocap_to_base_link_translation_m: [0.0, 0.0, 0.0]
  mocap_to_base_link_quaternion_xyzw: [0.0, 0.0, 0.0, 1.0]

stream:
  publish_zmq: "tcp://*:5556"
  output_pos_unit: "mm"
  output_vel_unit: "mm/s"
  quat_order_to_planner: "xyzw"

coordinate:
  source_frame: "hope_table"
  R_planner_from_source:
    - [1, 0, 0]
    - [0, 1, 0]
    - [0, 0, 1]
  t_planner_from_source_m: [-1.37, 0.7625, 0.02]

diagnostics:
  log_jsonl: true
  log_dir: "logs"
  print_interval_s: 1.0
```

刚体名称必须通过配置读取，不要硬编码 ID。
不要配置旧 UDP 桥接目标；Motive Unicast 目标由 Mac NatNet client 的握手和订阅决定。

## 五、第一阶段：只验证 NatNet 接收

先不要启用 ZMQ，也不要启动 Planner。

### 接收内容

每帧至少读取：

```text
frame_number
motive_timestamp
rigid body ID
rigid body position
rigid body quaternion
tracking_valid
mean_error（SDK 能提供时）
```

至少识别：

```text
table
ball
```

### 每秒输出一次

```text
receive_fps
current_frame
dropped_frames
drop_rate
timestamp_delta
table_valid
ball_valid
table_position
ball_position_world
```

不要每帧刷控制台。

### 成功标准

```text
FPS 接近 360
帧号持续增长
时间戳单调递增
table 持续有效
ball 静止时持续有效
drop 接近 0
```

### Mac 只能诊断的失败情况

如果没有数据：

```text
先检查 Mac 是否收到 UDP
```

建议输出诊断命令：

```bash
sudo tcpdump -ni <ethernet_interface> host 192.168.50.1 and udp
```

判断：

```text
有 UDP、程序无数据：
    NatNet 版本、解析器、客户端绑定地址或 Unicast 连接逻辑有问题

没有 UDP：
    Motive Streaming、Windows 防火墙、Local Interface 或网卡问题
```

后者只能让现场人员回 Windows 检查，Mac 不尝试远程修改。

## 六、第二阶段：建立球桌参考坐标

当前 Planner 使用球桌中心坐标系，不应直接使用 Motive 世界坐标。

每帧 NatNet 输入：

```text
table 世界位置：t_world_table
table 世界姿态：q_world_table
ball 世界位置：p_world_ball
```

基础转换：

```python
p_table = R_world_table.T @ (p_world_ball - t_world_table)
```

但是球桌是静止物体，**不建议每帧直接使用带噪声的 table 位姿计算球速度**。

推荐流程：

1. 启动后收集约 `300` 帧 table 位姿。
2. 对位置取均值。
3. 对姿态做稳定平均或选择稳定代表值。
4. 固定为本次运行的 `T_world_table_ref`。
5. 后续始终用固定参考变换处理球。
6. 实时 table 数据只用于监测球桌是否发生移动。

原因：

```text
如果每帧都使用轻微抖动的 table 位姿，
table 的姿态噪声会被传播到 ball position 和 ball velocity。
```

监控条件：

```text
table 相对初始位置移动超过阈值
table 姿态突然变化
table tracking invalid
```

发现后只报警，例如：

```text
TABLE_MOVED
TABLE_TRACKING_LOST
TABLE_ORIENTATION_FLIP
```

不要由 Mac 自动重建坐标系，避免运行中坐标突然跳变。

## 七、坐标方向验证

Planner 坐标约定：

```text
原点：球桌中心
+X：机器人一侧指向对方一侧
机器人位于 X 负方向
来球从对方飞向机器人时 vx < 0
+Z：向上
```

Planner 中：

```python
TARGET_HIT_X = -1.47
```

来球触发条件包括：

```python
vx < -1.0
-1.37 < x < 1.37
-0.7625 < y < 0.7625
0.0 < z < 1.0
```

现有 Planner 会把收到的 `pos` 和 `vel` 除以 `1000`，并通过本地 ZMQ `tcp://127.0.0.1:5556` 接收数据。

因此必须做人工验证：

```text
把球放在球桌中心：
    x ≈ 0
    y ≈ 0

沿球桌长轴向机器人移动：
    x 减小
    vx < 0

沿横向移动：
    主要改变 y

向上移动：
    z 增大
```

如果轴方向不符合，Mac 端通过配置增加固定轴变换：

```yaml
planner_axis:
  permutation: [0, 1, 2]
  signs: [1, 1, 1]
```

例如 X 方向相反时：

```yaml
signs: [-1, 1, 1]
```

不要直接改 Planner 阈值来掩盖坐标错误。

## 八、ball 刚体的特殊处理

当前 `ball` 是刚体，因此不用处理 Unlabeled Marker，也不用做最近邻关联。

流程：

```text
按名称找到 ball ID
→ 检查 tracking_valid
→ 获取 position / quaternion
→ 计算真实球心位置
→ 转换到 table frame
→ 计算速度
```

### 特殊问题 1：Pivot 可能不在真实球心

7 个 Marker 的几何中心未必等于球心。

现场测试：

1. 把球放在固定位置。
2. 缓慢旋转球，但尽量不移动球心。
3. 观察 `ball position` 是否画圈或明显漂移。

如果漂移明显，说明 Pivot 偏离球心。

Mac 端可以支持：

```yaml
ball:
  center_offset_local_m: [dx, dy, dz]
```

计算：

```python
p_ball_center_world = p_ball_rb_world + R_ball_world @ offset_local
```

注意：偏移必须在球刚体局部坐标系中旋转到世界坐标，不能简单加一个固定世界坐标偏移。

如果偏差很大，优先建议在 Windows Motive 中修正 Pivot；Mac 只提供临时补偿能力。

### 特殊问题 2：可见 Marker 数量变化

实际只能看到 3～4 个 Marker 时，重点监测：

```text
tracking_valid
mean_error
刚体丢失次数
位置突跳
四元数突跳
```

如果 tracking 无效：

```text
不发送该帧 ball 数据
不跨越丢失区间计算速度
```

恢复后的第一帧：

```text
只重置 position 和 timestamp
不输出新速度
```

### 特殊问题 3：对称 Marker 布局

如果 Marker 布局近似对称，可能出现刚体方向翻转。

虽然 Planner 不使用球姿态，但如果 Pivot 存在偏移，姿态翻转会让球心补偿产生跳变。

需要输出：

```text
BALL_ORIENTATION_JUMP
BALL_POSITION_JUMP
```

## 九、速度估计

不要直接使用普通两帧差分作为最终版本，因为 360 Hz 下位置噪声会被放大。

第一版可同时计算两种速度：

```text
raw_velocity
filtered_velocity
```

### 原始速度

```python
v_raw = (p_now - p_prev) / dt
```

仅用于诊断。

### 推荐 Planner 输入速度

使用最近 `5` 帧按真实时间戳做线性回归，分别拟合 X/Y/Z 的斜率。

优点：

```text
比两帧差分稳定
延迟较小
对不完全均匀时间间隔更可靠
```

要求：

* 使用 Motive timestamp。
* 不使用 `time.time()` 计算球速度。
* `dt <= 0` 时丢弃。
* 帧号不连续时记录 drop。
* tracking 恢复后重新积累窗口。
* 检测速度是否超过合理上限。

建议初始：

```text
window = 5 frames
max_speed = 20 m/s
```

所有阈值做成配置项。

## 十、第三阶段：判断 Planner 是否可运行

在接 ZMQ 前先检查 Mac 环境。

### 关键特殊问题：Planner 强依赖 ROS 2

当前 `predict_node_0627.py` 在启动时直接：

```python
import rclpy
rclpy.init(...)
create_node(...)
create_publisher(...)
```

它没有真正的 `--no-ros` 模式。当前 adapter 已经有 `--no-ros`，用于只验证 NatNet ball 数据源和 ZMQ 发布，不启动 ROS robot receiver。

因此 Codex 首先检查：

```bash
python -c "import rclpy; print('rclpy ok')"
```

如果失败，现有 Planner 在 Mac 上无法直接启动。

处理优先级：

```text
A. 如果 Mac 已有可用 ROS 2 环境：
   不修改 Planner，直接运行

B. 如果 Mac 没有 rclpy：
   adapter 先用 --no-ros 跑 ball-only 链路
   Planner 本体仍需要 ROS2/rclpy，除非另复制 predict_node_0627.py 做诊断版

C. 不要为了现场测试在 macOS 临时大规模安装或重构 ROS 2
```

这是开始接 Planner 前必须提前确认的问题。

### 机器人数据不是当前最小闭环的阻塞项

当前 Planner 初始化了默认机器人位置和四元数；在没有 `"type": "robot"` 消息时仍可运行，只是使用默认值。

因此第一轮只发送：

```text
type = ball
```

即可验证：

```text
发现新来球
首次锁定
时间更新
首次发布成功
事后校验
```

但这只能证明球轨迹预测链路，不能证明真实机器人坐标和 WBC 指令正确。

## 十一、ZMQ 接入

adapter 作为 Publisher：

```python
socket = context.socket(zmq.PUB)
socket.bind("tcp://127.0.0.1:5556")
```

Planner 已经是 Subscriber：

```python
socket.connect("tcp://127.0.0.1:5556")
```

球消息：

```python
{
    "type": "ball",
    "t": motive_timestamp_seconds,
    "pos": [x_mm, y_mm, z_mm],
    "vel": [vx_mm_s, vy_mm_s, vz_mm_s]
}
```

内部统一使用：

```text
m
m/s
s
```

只在发送前转换：

```python
pos_mm = pos_m * 1000.0
vel_mm_s = vel_m_s * 1000.0
```

### ZMQ 特殊问题

PUB/SUB 存在 slow joiner 现象。

建议启动顺序：

```text
先启动 adapter
等待 NatNet 稳定
再启动 Planner
```

adapter 在 ZMQ bind 后等待短暂时间再开始正式发送，或持续发送数据，不假设前几帧一定会被 Planner 收到。

## 十二、建议运行模式

### 模式 1：adapter ball-only

```bash
python3 -m mocap.hope_optitrack_adapter.main \
  --config mocap/hope_optitrack_adapter/config.yaml \
  --no-ros
```

只检查 Mac 是否能通过 NatNet Unicast 收到 `ball`，并通过 ZMQ `tcp://*:5556` 发布 ball 消息。控制台应每秒输出一行 `fps/latency_ms/valid_rate/drop`。

### 模式 2：adapter full

```bash
python3 -m mocap.hope_optitrack_adapter.main \
  --config mocap/hope_optitrack_adapter/config.yaml
```

在 Mac ROS2 环境可用、且 `motion_capture_tracking` 已发布 `/motion_capture_tracking/poses` 时使用。此模式会同时发布 robot 和 ball。

### 模式 3：planner

```bash
python3 planner/predict_node_0627.py
```

Planner 仍通过 `tcp://127.0.0.1:5556` 订阅，预测和击球逻辑不改。这样现场可以逐级定位问题，避免一启动就把 NatNet、坐标、速度、ZMQ、ROS 和 Planner 混在一起调试。
